# ADR-0017: Agentic OS Security Hardening and Core Primitives (Permission Layer, Memory, Artifacts, Coding Workspace)

Date: 2026-07-02
Status: accepted

## Context

A repo-wide audit (four parallel research passes over `packages/db`, `apps/server`, `apps/web`, and the security/plugin/skill surface) found the Agentic OS data model already substantial — `workspace`, `agent`, `task`, `agentRun`, `taskEvent`, `approvalRequest`, `auditLog`, `automation`, `plugin` all existed across both DB dialects, all wired to real tRPC procedures and real UI (task board, approvals page, run timeline, audit log, automations — functional, not stubs). Agent execution (`agent-runtime.ts`), one-level delegation (ADR-0011), the defer-and-resolve approval workflow (ADR-0009), and MCP tool gating all worked as designed.

The audit also found one disqualifying gap: **every tRPC procedure was `publicProcedure`** — `trpc/context.ts`'s `createContext` returned `{}`, no session was ever derived, and every one of ~160 procedures trusted a client-supplied `workspaceId`/entity id with no ownership check. Compounding this: a hardcoded fallback auth secret with no production guard, zero rate limiting anywhere, a CORS policy that reflected `allowedWebOrigins[0]` for any unrecognized origin instead of rejecting it, and — most consequential given ARCHITECTURE.md's explicit requirement that "full auto never means unbounded shell/filesystem/network/secrets" — `autonomous`/`super_agent` agents ran with `policy.mode: "auto"`, which bypassed the approval gate unconditionally, including for `terminal_run` and `file_delete`.

Given the scale of a full `packages/core-agent-engine` extraction (moving `agent-runtime.ts`, `tools.ts`, `delegation.ts`, `scheduler.ts` wholesale) against a large, working, already-tested system, and given "existing features must not break," this ADR documents a deliberate scope choice: fix the disqualifying security gap first, add the genuinely missing primitives (permission taxonomy, memory, artifacts) as new code in a new package rather than a risky move of working code, and defer the full engine extraction.

## Decision

**tRPC session auth.** `trpc/context.ts` derives a real better-auth session via `auth.api.getSession({ headers })` on every request (the web client already sends `credentials: "include"`, so this required no client-side transport change). `trpc/trpc.ts` adds `protectedProcedure` (requires a session) and `workspaceProcedure` (requires a session AND, via tRPC v11's `getRawInput()`, verifies the caller owns any `workspaceId` present in the raw input — a single shared middleware instead of hand-writing the check into every resolver). `trpc/workspace-guard.ts` adds `requireWorkspaceOwner` and `requireEntityWorkspaceOwner` (the latter for procedures that only take an entity id — task/agent/chat/etc — looking the entity up via the existing `DbRepository` getters, all of which return a `workspaceId`, and throwing `NOT_FOUND`/`FORBIDDEN` accordingly). Every procedure in `router.ts` was swept: 57 to `workspaceProcedure` (input already carries `workspaceId`), ~63 to `protectedProcedure` + an explicit ownership check (entity-id-only procedures, plus a handful of self-only checks like `workspaces.list`/`users.get`), and 15 left genuinely public (health, pre-login installation bootstrap, the `/share/{shareId}` viewer, static catalogs, a couple of host-wide CLI-auth endpoints gated to signed-in-only with a TODO since no ownership hook exists for them). The two plain (non-tRPC) Hono routes that bypassed this entirely — `POST /api/chat/stream` and the library upload/download routes — got the same session-plus-ownership check inline (`hono-auth.ts`), since they never went through `trpc/context.ts` in the first place.

Since the web app had **no login flow at all** before this (it only ever passed raw `userId`/`workspaceId` strings, no `better-auth/react` client, `installation.complete` created an account server-side without ever setting a browser session cookie), enforcing auth server-side without a client-side fix would have broken every page. `apps/web/src/lib/auth-client.ts` adds a `better-auth/react` client pointed at the same server the tRPC client uses; `AppShell` now gates all workspace-scoped content behind `useSession()` and renders `LoginScreen` (email/password) when no valid session exists; the setup wizard (`app/page.tsx`) calls `authClient.signIn.email` immediately after `installation.complete` succeeds so the owner's browser actually gets a session cookie instead of needing a separate manual login.

**Auth secret and CORS.** `auth.ts` throws at boot if `BETTER_AUTH_SECRET` is unset and `NODE_ENV === "production"` (dev keeps the convenience fallback). `index.ts`'s CORS `origin` callback returns `undefined` for an unrecognized origin instead of reflecting `allowedWebOrigins[0]`, so no `Access-Control-Allow-Origin` header is sent at all for a request from outside the allowlist.

**Rate limiting.** `rate-limit.ts` is an in-memory per-key sliding-window limiter (no Redis dependency, consistent with PC mode having no external services running — same reasoning as ADR-0010's DB-backed scheduler) applied to `/api/auth/*` (tighter budget, brute-force protection) and `/trpc/*` (general budget).

**Autonomous-bypass cap.** `tools.ts` adds `ALWAYS_REQUIRES_APPROVAL_KINDS` (`terminal_run`, `terminal_send_input`, `terminal_kill`, `task_run`, `test_run`, `custom_code`, `file_delete`, `browser_run_playwright_code`) — `shouldDeferToolForApproval` now checks this set *before* the `policy.mode === "auto"` early-return, so these categories stay gated even for `autonomous`/`super_agent` agents. This directly satisfies ARCHITECTURE.md's "full auto never means unbounded shell/filesystem access" — MCP tool calls are deliberately left out of this cap for now (documented as a follow-up: MCP has no per-tool network/side-effect classification to cap selectively, and capping all MCP calls in auto mode would break legitimate read-only autonomous MCP use).

**Permission taxonomy.** New `packages/core-agent-engine` package hosts `PermissionCategory` (`file.read/write/delete`, `repo.read/write`, `terminal.run`, `network.fetch`, `browser.use`, `mcp.call`, `plugin.execute`, `skill.execute`, `secret.read`, `automation.create`, `email.send`, `calendar.write`, `database.write` — the exact taxonomy from the plan) with an exhaustive `TOOL_KIND_PERMISSION` map from the existing `ToolKind` union (a TypeScript compile error if a new `ToolKind` is added without classifying it, not a silent fallback), a `PermissionRisk` (low/medium/high) per category, `hashToolInput` (canonicalized-JSON SHA-256, so semantically-identical calls hash identically regardless of key order), and `buildPermissionSnapshot`. `tools.ts`'s `buildToolsForAgent` and `approvals.ts`'s `resolveApprovalDecision` both route every audit-log write through this, populating two new `audit_log` columns (`inputHash`, `permissionSnapshot` — both dialects, additive migration) so "what was actually checked, and against which policy" survives later changes to an agent's autonomy level.

**Approval-request enrichment.** `approval_request` gains `title`, `description`, `riskLevel`, `affectedResources` (string array), `diffPreview` — all nullable/additive — matching the plan's approval-dialog requirements (title/description/risk/diff preview). The Approvals page renders these when present, falling back to the raw tool label/JSON input when a caller hasn't populated them yet (no caller currently populates them — wiring per-tool title/diff generation into `tools.ts`'s three defer sites is follow-up work, tracked, not done in this pass).

**Memory and Artifacts.** New `memory_entry` table (type/source/confidence/createdBy/expiry, matching the plan's `MemoryType` taxonomy exactly) and `artifact` table (type/title/content, linked to `task`/`agentRun`/`agent`), both dialects, with `DbRepository` methods and a `memory`/`artifacts` tRPC router (workspace-scoped list/create, entity-scoped update/delete with ownership checks). A `/workspace/[id]/memory` page provides create/delete UI grouped by type; the task detail page gained an Artifacts card. Nothing currently *writes* an artifact automatically from an agent run — the table, repo methods, and read UI exist; wiring a `create_artifact`-style tool or auto-capturing tool output as an artifact is follow-up work.

**Coding workspace groundwork.** `coding.ts` adds three read-only operations — `getRepoInfo`/`getGitStatus`/`getGitDiff` (via `Bun.spawn(["git", ...argv])`, never a shell string, so there's no injection surface from a repo/file path) and `listDirectory` (via skills-sdk's existing permission-scoped `readDir`, confined to the given root). A new `coding` tRPC router and `/workspace/[id]/coding` page (repo path picker reusing the existing `WorkingDirectoryPicker`, lazy-expanding file tree, git status list, diff viewer) give a first real Coding-workspace surface. Deliberately read-only end to end: no patch-apply action exists on this page yet — actual writes still go through the existing approval-gated file tools, matching the plan's "diff-first" principle. Wiring an in-page "apply this patch" action that creates a real approval request is follow-up work.

## Consequences

- The tRPC auth sweep is the single highest-leverage fix in this pass — it was the one gap that undermined every other control (the approval workflow, MCP gating, workspace scoping all assumed a workspaceId could be trusted, which it never could be until now).
- `packages/core-agent-engine` today holds only new, additive primitives (permission taxonomy, memory/artifact types) — it is explicitly **not** yet the home for `agent-runtime.ts`/`tools.ts`/`delegation.ts`/`scheduler.ts`, which remain in `apps/server` as working, tested code. A full extraction is a real, separate, higher-risk piece of work and should get its own ADR when undertaken rather than being bundled into a security-hardening pass.
- Skill/plugin code execution sandboxing (ADR-0007's flagged gap) is **unchanged** by this pass — still in-process, permission-checked-but-bypassable, still safe only under the "first-party trusted code" assumption ADR-0007 documents. This ADR does not claim to have addressed it; it remains the correct blocker before installing untrusted third-party plugins by default.
- A parallel, documentation-only audit stream (see `docs/SECURITY_AUDIT.md`, `docs/AGENTIC_OS_BACKLOG.md`, `docs/PLUGIN_SECURITY.md`, `docs/DEPLOYMENT_HARDENING.md`, `docs/TESTING_STRATEGY.md`, `docs/CI_QUALITY_GATES.md`) ran concurrently on the same branch and independently catalogued further gaps this pass did not address: encrypting secrets at rest (model provider API keys currently stored plaintext), not returning plaintext API keys to the browser, request body size limits, rate-limiting the library-upload/chat-SSE routes specifically, Caddy header hardening (HSTS/CSP), non-root Docker users, and process/worker isolation for skills and plugins (the ADR-0007 gap above). That backlog is the authoritative next-steps list for security work beyond this ADR's scope — see it directly rather than this ADR re-deriving it.
- Approval dialogs can now display title/description/risk/diff-preview, but nothing populates those fields yet (they're always null from every current call site) — the UI change is real but has no visible effect until a follow-up wires per-tool-kind title/diff generation into `tools.ts`.
- Two pre-existing, unrelated issues were found and left alone rather than fixed as drive-bys: a non-hermetic `seo-analyzer.test.ts` test that picks up whatever local model provider happens to be running on the dev machine (also independently found by the parallel audit as BL-13), and a `radix-ui@1.6.1` type-definition gap (`DropdownMenuContent` lacks `onOpenAutoFocus` in this version) that fails `next build`'s type-check step in `chat-top-bar.tsx` — fixing the latter requires a product decision (drop the auto-focus-search behavior, or pin a different radix version) outside this ADR's mandate.
