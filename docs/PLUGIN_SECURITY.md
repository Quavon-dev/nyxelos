# Plugin & Skill Security Model

Companion to [`SECURITY_AUDIT.md`](SECURITY_AUDIT.md) (see SEC-03, SEC-08, SEC-09) — this document is the fuller threat model for NyxelOS's three extensibility surfaces: **skills**, **plugins**, and **MCP servers** (`docs/ARCHITECTURE.md` section 8). It describes what's actually enforced today (not what's aspirational — see the doc/code mismatches flagged below), and a staged plan to close the gap, without requiring the Plugin/Skill Runtime rewrite this audit is explicitly not doing.

## What runs, and where

| Source | Defined by | Runs as | Permission enforcement |
|---|---|---|---|
| Builtin skills | `packages/skills-sdk/src/skills/*.ts`, hand-written, shipped with the app | In-process, same as server code | N/A — trusted first-party code |
| Custom skills | Workspace `skill` table row (kind + JSON config), created via the Skills tab, no restart needed | In-process, dynamically built by `apps/server/src/skills-resolve.ts` (config-driven: HTTP fetch, file read/write/list, KB search) — **except** the `custom_code` kind, which runs in an isolated subprocess (see stage 5 below) | `SkillPermissions` checked via `packages/skills-sdk/src/runtime.ts` |
| Plugin skills | Downloaded from an arbitrary GitHub repo (`apps/server/src/plugins.ts`), `SKILL.md`/`skills/*/SKILL.md` files parsed and loaded as `SkillDefinition`s | In-process, same runtime as above | Same `SkillPermissions` check, IF the skill only uses `ctx.*` |
| MCP tools | A connected MCP server (catalog entry in `apps/server/src/mcp-connectors.ts`, a custom server, or a local `stdio` process) | **Out-of-process** — either a remote HTTP endpoint or a locally spawned subprocess (`command`/`args` on the `mcpServer` record) | tRPC/approval-layer gating only; the MCP server itself is a black box once connected |

## The core gap: permission profile is advisory, not enforced, for skill/plugin code

`packages/skills-sdk/src/runtime.ts` builds a `SkillContext` whose `readFile`/`writeFile`/`fetch`/etc. are checked against the skill's declared `SkillPermissions` (`assertPathAllowed`, `hostAllowed`). This works correctly — verified in this audit with 8 new tests in `packages/skills-sdk/src/runtime.test.ts` (path traversal, absolute-path escape, sibling-directory-prefix trick, and host-allowlist substring trick all correctly rejected).

**But it only constrains code that calls through `ctx.*`.** A skill's `run(input, ctx)` function is plain TypeScript executing in the same process, with the same access to `process.env`, `require("node:fs")`, `require("node:child_process")`, global `fetch`, etc. as the server itself. Nothing stops a skill from ignoring `ctx` entirely. The runtime's own source comment says this directly: *"it is not a security boundary against a deliberately malicious skill... Process- or container-level isolation is tracked as a follow-up; see ADR-0007."*

`docs/ARCHITECTURE.md` section 12 currently says skills "run in isolated worker processes" — that line describes the ADR-0007 target, not the current implementation, and should be corrected or clearly marked "planned" (see `SECURITY_AUDIT.md` SEC-03 / backlog `BL-03`).

## Plugin install pipeline — what is and isn't checked

`installPluginFromGithub()` in `apps/server/src/plugins.ts`:

1. Parses an `owner/repo[/tree/ref]` string from user input (`parseGithubRepoUrl` — well-tested, `plugins.test.ts`).
2. Resolves the default branch if no ref given, fetches the full repo tree via the GitHub REST API (unauthenticated — subject to GitHub's public rate limit, ~60 req/hr per IP, shared across all workspaces on that server).
3. Downloads every blob under `MAX_FILE_BYTES` (25MB), preserving folder structure, into `PLUGINS_ROOT/<workspaceId>/<slug>/`.
4. Registers every `skills/<name>/SKILL.md` (or root `SKILL.md`) as a loadable skill, parses `agents/*.md` as display-only sub-agent definitions.

**Checked:** repo tree size (GitHub truncation error surfaced), per-file size, that the URL parses as a GitHub repo.
**Not checked:** author/maintainer identity or reputation, commit signature, file contents against any allowlist/denylist, whether `SKILL.md` bodies contain code that reaches outside `ctx.*` (see above), whether the same slug was previously installed by a different, unrelated repo (slug collision is handled — `slugify()` + reinstall-replaces-existing — but that's a UX safeguard, not a security one).

This matches the project's own roadmap (`ARCHITECTURE.md` Phase 6: *"Remaining: ... signed plugin manifests"*) — it is a known, tracked gap, not a surprise. This document exists so the gap is visible in the security-specific docs too, not just the phased roadmap.

## Threat scenarios

1. **Malicious plugin author.** A user pastes a GitHub URL for a plugin that looks legitimate (README, screenshots, a real `SKILL.md`) but whose `run()` function reads `process.env` and `fetch()`s it to an attacker-controlled URL, or shells out via `Bun.spawn`/`child_process`. Nothing in the install or load pipeline would catch this. Mitigated only by the approval workflow if the skill is marked `sensitive: true` — but the plugin author controls that flag too (it's part of the skill definition they ship).
2. **Compromised upstream repo.** A previously-trustworthy plugin repo is compromised (maintainer account takeover, malicious PR merged) and a user reinstalls/updates — `installPluginFromGithub` always re-downloads and replaces on reinstall, so there's no version pinning protecting an existing install from picking up a newly-malicious update on next reinstall. (A *first* install pins to a ref only if the user explicitly used `/tree/<ref>` in the URL; the default is "current default branch," which moves.)
3. **MCP server as a black box.** A custom or catalog MCP connector, once connected, can offer any tool with any name/description at call time — the client (`packages/mcp-client`) trusts what the server advertises. This is inherent to the MCP protocol as currently specified and not specific to NyxelOS, but worth noting: the "same permission dialog before activation" (`ARCHITECTURE.md` section 8) is a one-time approval of *connecting*, not a per-tool-call permission profile the way skills have.

## Staged mitigation plan (does not require Plugin/Skill Runtime rewrite)

These are ordered cheapest-and-safest first; none require the isolation work in ADR-0007 to land first.

1. **Correct the docs** (`ARCHITECTURE.md` section 12, this file) to state plainly that the permission profile is enforced for `ctx.*`-based access only, not a sandbox. Zero code risk, immediate. → backlog `BL-03`. **Done** — section 12 rewritten.
1.5. **Kill switch (done, this session).** Rather than wait for stages 2–5, both entry points for unsandboxed code execution are now disabled by default in production: `installPluginFromGithub` (`apps/server/src/plugins.ts`) refuses to run unless `ENABLE_REMOTE_PLUGIN_INSTALL=true`, and the `custom_code` tool kind (`apps/server/src/tools-dynamic.ts`) refuses to run unless `ENABLE_CUSTOM_CODE_SKILLS=true`. Both default to enabled outside production (`NODE_ENV !== "production"`) so local dev is unaffected. See `apps/server/src/feature-flags.ts` + `feature-flags.test.ts`. This is coarser than stages 2–4 below (all-or-nothing per server, not per-plugin) but closes the "silently on in prod" gap immediately without touching the plugin/skill runtime.
2. **Warn at install time.** The Plugins page install dialog could surface a static, non-blocking warning: "Plugins run with the same access as the server itself; only install plugins from sources you trust." This is a UI copy change, low risk, high value given #1. **Done** — the install card on the Plugins page (`apps/web/src/app/workspace/[workspaceId]/plugins/page.tsx`) always shows a static warning: plugins can run code with full server access, only install from trusted sources, prefer a commit SHA/tag over a moving branch, and review contributed skills/permissions before relying on it. This is copy only — it doesn't gate anything by itself (see #4 for the part that does).
3. **Ref-pin by default.** When no `/tree/<ref>` is given, resolve and store the *commit SHA* (not just branch name) at install time, and only move forward on an explicit "check for updates" action rather than silently re-resolving the default branch on every reinstall. Reduces threat scenario #2 to an explicit, auditable user action. **Partially done** — `installPluginFromGithub` (`apps/server/src/plugins.ts`) now best-effort-resolves the commit SHA for whatever ref it uses (explicit `/tree/<ref>` or the default branch) and stores `ref`/`resolvedSha`/`refPinned` on the `plugin` row. `refPinned` is only true when the user pastes an exact 40-character commit SHA via `/tree/<sha>`; a branch or tag name (explicit or defaulted) is flagged in the UI as "not pinned" and still moves on the next reinstall. **What's still open:** there is no separate "check for updates" action yet — reinstalling (pasting the same URL again) is still the only way to update, and it re-resolves the ref from scratch rather than diffing against the previously-installed commit.
4. **Static scan on install.** Before registering a downloaded `SKILL.md`'s code as runnable, grep the file for high-risk identifiers (`child_process`, `process.env`, `require(`, `eval(`, raw `fs.` outside the `ctx` pattern) and surface a warning listing what was found, without blocking install. Cheap, no runtime cost, no false-negative-free but raises the bar for casual/careless plugins. **Done** — `scanForRiskyPatterns` in `apps/server/src/plugins.ts` regex-scans every downloaded text file (skipping known binary extensions and files over 2MB) for `process.env`, `child_process`, `fs.rm`/`fs.unlink`, raw `fetch(...)` calls (an allowlist can't be verified statically, so any `fetch(` is flagged, not just unsafe ones), `Bun.spawn`, and `eval(...)`/`new Function(...)`. If anything is found, the install is **not silently blocked**: `installPluginFromGithub` throws `PluginInstallNeedsConfirmationError` carrying the full finding list, nothing is written to disk or the DB, and the `plugins.install` tRPC procedure turns that into a `{ status: "needs_confirmation", riskSummary }` response. The Plugins page shows a confirmation dialog listing every finding; the user must explicitly click "Install anyway" (which resubmits with `acknowledgeRisk: true`) before anything installs. Findings that were acknowledged are also persisted on the `plugin` row (`riskFindings`) so they stay visible after install, not just at the moment of confirmation. **This is still a naive substring/regex scan, not static analysis** — a clean result never means "safe" (trivially evadable by a determined author, e.g. string concatenation or indirection), and a flagged result can just as easily be legitimate code (a doc example, a comment). It raises the bar against casual/careless plugins; it is not, and is not presented as, a security boundary.
5. **Real process/container isolation (ADR-0007's real fix).** Longer-term: run skill/plugin `run()` calls in a Bun `Worker`, a short-lived subprocess, or a container, with its own restricted environment (only the env vars the skill's declared permissions justify) and IPC-based `ctx.*` calls back to the main process. This is the only item on this list that actually closes the gap rather than raising the bar — it's also the only one that's a genuine Skill Runtime change, correctly out of scope for anything short of a dedicated design effort. **Partially done** — the `custom_code` tool kind (`apps/server/src/tools-dynamic.ts`), the one place a user-supplied JavaScript *string* is executed (as opposed to a skill's own hand-written/markdown-defined `run()`), now runs through `apps/server/src/plugin-sandbox.ts`: a fresh subprocess per call, spawned with a caller-controlled `env` (never the server's real `process.env`), running the code inside a `node:vm` context whose only globals are `input` and `ctx` — no `require`, `process`, `Bun`, or any other ambient API. `ctx` is rebuilt inside the subprocess from `createSkillContext` using the tool's own declared `permissions` (never the server's secrets), so filesystem/network access still goes through the same allow-list checks as every other skill. See `apps/server/src/plugin-sandbox.test.ts` for the regression tests (process.env isolation, `require`/`Bun` unavailability, permission enforcement, timeout).

   **Extended this session** to the other place in the codebase that ran a user-supplied JavaScript string unsandboxed: `browser_run_playwright_code` (`apps/server/src/tools-builtin/browser.ts`), which previously ran arbitrary code via a raw `new Function("page", ...)` in-process — same threat model as pre-ADR-0007 `custom_code`, just against a live Playwright `Page` instead of a serializable `ctx`. Since a live `Page` handle can't cross a process boundary the way `ctx`'s permission-checked calls can, this couldn't reuse `runIsolatedCustomCode`'s subprocess directly; `plugin-sandbox.ts` instead gained a second, narrower primitive, `runVmSandboxedCode(code, globals, options)` — a same-process `node:vm` context whose only globals are whatever the caller explicitly passes in (here, just `page`), with the identical no-`require`/`process`/`Bun`/ambient-`fetch` guarantee, a timeout, and the same output/error size cap as `runIsolatedCustomCode`. `buildBrowserRunPlaywrightCodeTool` now calls `runVmSandboxedCode(code, { page })` instead of `new Function(...)`. This is honestly weaker than the subprocess path — no process/env boundary of its own (there's nothing secret in `globals` to leak, but a hung script inside an awaited call keeps running in the background past the timeout, since there's no separate process to kill) — see `runVmSandboxedCode`'s own doc comment in `plugin-sandbox.ts` for the exact trade-off. It still closes the concrete gap that mattered: the sandboxed code can no longer read `process.env`, `require("node:fs")`/`child_process`, or reach the `Bun` global, only raised bar over `new Function`. See `apps/server/src/plugin-sandbox.test.ts` (mechanism-level tests against a plain stand-in object) and `apps/server/src/tools-builtin/browser.test.ts` (end-to-end against a real headless Chromium page) for the regression tests.

   **What's still open:** everything *else* — hand-written builtin skills, DB-backed tools of every other kind, and plugin-contributed `SKILL.md` bodies — still runs in-process with the same access as the server itself, unsandboxed (though, per the file-skill investigation this session, plugin-contributed `SKILL.md`/agent files are purely declarative text handed back to the model today — they don't execute as code at all yet, so there is currently no live gap there to close, only the plugin *install* pipeline's static risk-scan of files that aren't themselves executed). The subprocess/vm paths above are also still a full Bun runtime outside their own `vm` context, not a kernel/container boundary — a `vm`-escape is a known theoretical risk class neither claims to close, only substantially raise the bar over same-process `new Function`.
