# Testing Strategy

## Current state

```
bun test
```
from the repo root passes cleanly on a fresh checkout — no dead files, no non-hermetic failures. `bun run test` (root `package.json`) is the documented entry point, matching the `dev`/`build`/`typecheck`/`lint` pattern, and `ci.yml`'s `test` job runs it after `lint`/`typecheck` on every PR (see [`CI_QUALITY_GATES.md`](CI_QUALITY_GATES.md)).

### History: the 5 dead files and 2 non-hermetic failures this section used to describe

An earlier audit found `bun test` reporting 77 pass/7 fail/5 errors: 5 dead `/tests/*.test.ts` files left over from before the current monorepo layout (importing paths like `../services/db` that no longer exist), plus 2 `seo-analyzer.test.ts` failures traced to `pickBestModelIdForSeo` falling through to a real local-model-detection probe instead of a seeded/mocked model list. A later pass deleted the 5 dead files, added the root `test` script and CI job, and fixed the `pickBestModelIdForSeo` fallback logic itself.

### A second, related hermeticity bug found and fixed in this pass

Deleting the dead files and fixing `pickBestModelIdForSeo` wasn't quite the whole story: `apps/server/src/plugins.test.ts`, `skills-resolve.test.ts`, and the rest of `seo-analyzer.test.ts` called `getDb()` directly with no test-DB setup at all, relying on whatever `DATABASE_URL` resolves to by default (`apps/server/nyxel.sqlite`) already existing and being migrated. On a developer machine that had already run the app (`bun run db:migrate` at some point, or just started the dev server once), this silently passed. On a genuinely fresh checkout — which is exactly what CI runs — that file either doesn't exist yet or exists with no tables, and every test in those files failed with `SQLiteError: no such table: user`. Fixed by wrapping each of those files in `installTestDb()`/`beforeEach`/`afterEach` (`packages/db/src/test-utils.ts`), the same throwaway-temp-SQLite-with-real-migrations pattern `agent-runtime.test.ts`/`goal-orchestrator.test.ts`/`knowledge-base.test.ts` already used. **Takeaway for future test files:** never call `getDb()` (or anything that transitively does) without first calling `installTestDb()` in a `beforeEach` — a test that happens to pass because the ambient dev DB is already migrated is not actually hermetic, and will fail the moment CI (or a teammate's fresh clone) runs it.

## What "isolated test" means for this repo, going forward

The pattern already established and worth continuing (`plugins.test.ts`, `skills-resolve.test.ts`, `workspace-file-tools.test.ts`, and this audit's `runtime.test.ts`): pure-function and permission-boundary tests that use real temp directories (`mkdtemp`) rather than mocking the filesystem, and fake/injected dependencies (see `plugins.test.ts`'s fetch stubbing) rather than hitting real network/DB where avoidable. Tests that must touch a real DB appear to use an in-memory/temp SQLite instance per test (worth confirming and documenting explicitly here once the core team's convention is settled — this audit didn't want to presume one).

## Reliability foundation pass — new coverage

New tests added alongside the reliability-hardening changes documented in `ARCHITECTURE.md` section 12a:

- **Permission matrix** — `packages/core-agent-engine/src/permissions.test.ts` exercises `resolveToolPermission()` across every source (skill/tool/mcp/workflow/delegation), confirming `ALWAYS_REQUIRES_APPROVAL_KINDS` entries always require approval, `custom_code` is flagged `sandboxRequired`, and MCP tools default to `requiresApproval: true`.
- **Typed pause outcomes / no text-marker detection** — `packages/core-agent-engine/src/run-outcome.test.ts` covers `resolveRunOutcome()`'s question > approval > budget priority and includes an explicit regression case proving it never inspects output text (a budget-blocked run whose reason string happens to contain `"pending_approval"` is still classified as a budget pause).
- **Delegation depth and cycle protection** — `packages/core-agent-engine/src/delegation-policy.test.ts` (pure `canDelegateDeeper`/`filterCyclicCandidates` logic), `apps/server/src/delegation.test.ts` (real-DB test proving `buildDelegateToAgentTool` drops a candidate already in the delegation chain), and `apps/server/src/tools.test.ts` (proving `delegate_to_agent` is hidden entirely once `MAX_DELEGATION_DEPTH` is reached).
- **Durable execution fields** — `packages/db/src/repo/agent-run-retry.test.ts` round-trips `retryCount`/`maxRetries`/`nextRetryAt` through `createAgentRun`/`updateAgentRun` and confirms `dead_letter` is a valid terminal status excluded from "active" runs.
- **Sandbox output cap** — `apps/server/src/plugin-sandbox.test.ts` gained cases for the new `maxOutputChars` cap on both a sandboxed call's return value and a thrown error message.
- **Mockable model execution / agent-runtime reliability** — the "needs a mockable model-call injection point" gap above is closed: `agent-runtime.ts` now calls every real model request (planning, direct execution, delegation synthesis/verifier) through a single swappable `streamChatImpl` reference, defaulting to the real `streamChat` from `@nyxel/model-providers` in production; `__setStreamChatImplForTests`/`__resetStreamChatImplForTests` let a test substitute a fake implementation. `apps/server/src/agent-runtime.test.ts` adds a `makeScriptedStreamChat` harness (a fake `streamChat` that plays back a queue of scripted "turns" — plain text, real tool calls that invoke the actual `execute()` closure from the run's own `ToolSet`, or a thrown error — falling back to its last turn once exhausted, since a retried attempt re-runs planning from the top) plus a `makeHangingStreamChat` helper (never resolves except via the caller's `abortSignal`, for cancellation). New coverage: planning → direct execution → completion; a real tool call persisted to the task-event timeline; approval pause → approve → resume and → reject → blocked (using a real `file_delete` tool and `resolveApprovalDecision`, not a simulated approval); an autonomy-budget pause mid-run (`maxToolCallsPerRun`); transient-failure retry-then-succeed and retry-exhausted-then-`dead_letter`; a non-transient failure with no retry; cancellation of a run that's mid-model-call; and delegation depth-limit/cycle-prevention (both fall back to direct execution instead of recursing). None of these hit a real local/cloud model or spawn a CLI subprocess.

Secret redaction and path-traversal prevention were already fully covered (`audit.test.ts`, `skills-sdk/src/runtime.test.ts`) before this pass — the new secret-redaction call site (task-event payloads, `agent-runtime.ts`) reuses the already-tested `sanitizeForAudit()` rather than adding a parallel implementation, so no new test was needed there.
