# Testing Strategy

## Current state

```
bun test
```
from the repo root passes cleanly on a fresh checkout ‚Äî no dead files, no non-hermetic failures. `bun run test` (root `package.json`) is the documented entry point, matching the `dev`/`build`/`typecheck`/`lint` pattern, and `ci.yml`'s `test` job runs it after `lint`/`typecheck` on every PR (see [`CI_QUALITY_GATES.md`](CI_QUALITY_GATES.md)).

### History: the 5 dead files and 2 non-hermetic failures this section used to describe

An earlier audit found `bun test` reporting 77 pass/7 fail/5 errors: 5 dead `/tests/*.test.ts` files left over from before the current monorepo layout (importing paths like `../services/db` that no longer exist), plus 2 `seo-analyzer.test.ts` failures traced to `pickBestModelIdForSeo` falling through to a real local-model-detection probe instead of a seeded/mocked model list. A later pass deleted the 5 dead files, added the root `test` script and CI job, and fixed the `pickBestModelIdForSeo` fallback logic itself.

### A second, related hermeticity bug found and fixed in this pass

Deleting the dead files and fixing `pickBestModelIdForSeo` wasn't quite the whole story: `apps/server/src/plugins.test.ts`, `skills-resolve.test.ts`, and the rest of `seo-analyzer.test.ts` called `getDb()` directly with no test-DB setup at all, relying on whatever `DATABASE_URL` resolves to by default (`apps/server/nyxel.sqlite`) already existing and being migrated. On a developer machine that had already run the app (`bun run db:migrate` at some point, or just started the dev server once), this silently passed. On a genuinely fresh checkout ‚Äî which is exactly what CI runs ‚Äî that file either doesn't exist yet or exists with no tables, and every test in those files failed with `SQLiteError: no such table: user`. Fixed by wrapping each of those files in `installTestDb()`/`beforeEach`/`afterEach` (`packages/db/src/test-utils.ts`), the same throwaway-temp-SQLite-with-real-migrations pattern `agent-runtime.test.ts`/`goal-orchestrator.test.ts`/`knowledge-base.test.ts` already used. **Takeaway for future test files:** never call `getDb()` (or anything that transitively does) without first calling `installTestDb()` in a `beforeEach` ‚Äî a test that happens to pass because the ambient dev DB is already migrated is not actually hermetic, and will fail the moment CI (or a teammate's fresh clone) runs it.

## What "isolated test" means for this repo, going forward

The pattern already established and worth continuing (`plugins.test.ts`, `skills-resolve.test.ts`, `workspace-file-tools.test.ts`, and this audit's `runtime.test.ts`): pure-function and permission-boundary tests that use real temp directories (`mkdtemp`) rather than mocking the filesystem, and fake/injected dependencies (see `plugins.test.ts`'s fetch stubbing) rather than hitting real network/DB where avoidable. Tests that must touch a real DB appear to use an in-memory/temp SQLite instance per test (worth confirming and documenting explicitly here once the core team's convention is settled ‚Äî this audit didn't want to presume one).
