## 1. Vision and Guiding Principles

NyxelOS is a fully open source operating system for AI agents that runs entirely on your own computer or your own server and never sends data to third parties unless the user deliberately connects an external model provider. The name is a short, invented, infrastructure-sounding word with no literal meaning attached â€” chosen deliberately as a clean, brandable product name rather than a descriptive compound. A collision check found no existing open source projects, products, or notable accounts under this name, while more literal options like "AgentOS" are already taken by several projects (e.g. agentos.org, The-Swarm-Corporation/AgentOS, rivet-dev/agentos), and other invented candidates considered along the way (NyxelOS, Vantrix, Halvex, Draveon) were either dropped in favor of a cleaner sound or turned out to collide with existing small projects. If "NyxelOS" turns out to collide during a final domain/trademark check, "Quorin" or "Ilvara" are backup names â€” both also came up empty in research.

Four principles run through every decision in this plan. First, local-first: the core functionality must work without an internet connection and without external accounts once a local model is installed. Second, modularity: models, skills, MCP servers, plugins, and even the database are swappable building blocks behind clear interfaces, never hard-wired. Third, security by default: every autonomy level an agent has must be explicitly granted; nothing should silently gain more permissions than the user allowed. Fourth, one consistently applied design: shadcn/ui in its default theme, exclusively the default color tokens from globals.css, no custom colors, no deviation from the component library.

## 2. Architecture Overview

NyxelOS consists of five major layers. The web UI (Next.js App Router, TanStack Query, shadcn/ui) talks exclusively to the backend server, never directly to models or external services. The backend server (Bun runtime, Hono as the HTTP layer, tRPC for typed requests, dedicated SSE routes for streaming) handles authentication, data storage, and the actual agent engine. The agent engine orchestrates chats, autonomous agents, and super-agents, picks models from the model provider layer, and calls tools through the MCP client. The MCP layer is bidirectional: NyxelOS connects as a client to any MCP server (Obsidian, filesystem, browser, your own Mac) while also exposing its own MCP server, through which skills can be offered outward (e.g. to Claude Desktop or Claude Code). The fifth layer is the local companion helpers â€” small native processes on the respective operating system that expose calendar, contacts, and photos without the NyxelOS server itself needing OS-native permissions.

This separation enables exactly the two required deployment modes: on your own PC, everything including the database runs in a single process or Docker Compose stack without internet access; on your own server, the same server container runs behind a reverse proxy with a real domain and login, without any code changes.

## 5. Domain Model

At the center is the workspace, a category into which users sort their chats, agents, and automations (e.g. "Work", "Personal", "Coding", "Research"). Each workspace has its own custom instructions, which are prepended as a system-prompt block before every conversation, as well as its own selection of enabled skills and MCP servers. A chat is a single conversation with a chosen model; an agent is a saved configuration of system prompt, model, assigned skills/MCP tools, and an autonomy level. Autonomy levels are tiered: "Chat" only responds on request, "Assisted" may call tools but must have the user confirm any security-relevant action (deleting a file, sending a message, triggering a purchase), "Autonomous" runs on a schedule or in response to events in the background within previously granted limits, and "Super-agent" can additionally delegate to other agents as subtasks and merge their results. Every action by every agent is logged in the audit log, including the model used, the tool called, and the result.

## 6. The Agent System in Detail

Normal chats are the simplest case: a user types, the chosen model responds with streaming, optionally using individual skills or MCP tools the user has enabled for that chat. Autonomous agents run without active input, triggered either by a schedule (e.g. "summarize today's calendar every morning at 7am") or by an event (e.g. "when a new file lands in the Downloads folder"). They have a tightly scoped tool and network budget defined at creation time, and can request approval before performing an irreversible action. Super-agents are orchestrators: they break a complex request into subtasks, assign each subtask to a specialized sub-agent or skill, collect the intermediate results, and merge them â€” conceptually similar to a project lead coordinating specialists, except that each "specialist" here can be its own configured agent with its own model and tools.

## 7. Model Provider Layer

NyxelOS distinguishes three provider categories, all addressed through the same Vercel AI SDK abstraction. Local models are auto-detected on startup by having the server probe common local ports: Ollama at `localhost:11434/api/tags`, LM Studio and other OpenAI-compatible servers at `/v1/models`. Detected models appear automatically in the model picker with no manual configuration. Cloud API models (Anthropic, OpenAI, Google, others) are connected via stored API keys, which are held encrypted in the database. As a third category, any OpenAI-compatible endpoint can be added manually, which also covers self-hosted third-party models or data-center clusters. For every model, the UI shows context window size, whether it runs locally or externally, and for API models, a rough cost estimate per request.

## 8. Skills, Plugins, and MCP Servers

Skills are small, self-contained capabilities (e.g. "summarize a PDF," "create an Obsidian note") that run in a sandboxed runtime and carry a clearly declared permission profile (filesystem access, network access, which directories). As of ADR-0013, skills come from three sources merged into one catalog: a small process-wide set of hand-written "builtin" skills (`packages/skills-sdk`), workspace-scoped "custom" skills defined declaratively (kind + JSON config: HTTP fetch, file read/write/list, knowledge-base search, or a short custom-code function) through the workspace's Skills tab and stored in the `skill` table â€” no server restart or TypeScript required to add one â€” and, as of ADR-0014, skills contributed by an installed **plugin**. Plugins are larger extensions that bundle multiple skills, sub-agents, and arbitrary supporting files (scripts, references, docs) â€” comparable to an app-store listing. Installing one (`/workspace/[id]/plugins`, paste a GitHub repo URL) downloads the repo's full folder structure verbatim rather than flattening it to a single skill body, matching the Claude Code plugin format (`.claude-plugin/plugin.json` + `skills/` + `agents/`) with a generic fallback for repos that don't use it. MCP servers are the third, complementary building block: instead of building every integration from scratch, NyxelOS connects to the growing MCP ecosystem (per the official MCP registry and community directories, now tens of thousands of available servers), for example for GitHub, filesystem, browser automation, or Obsidian itself. The MCP specification finalizing in July 2026 introduces a stateless protocol core, caching metadata for tool listings, and stricter authorization, all of which the MCP client layer is designed around from the start. Every skill, plugin, and MCP server goes through the same permission dialog before activation, which the user must explicitly confirm; skills marked sensitive (writes, custom code, and every MCP tool) additionally go through the approval workflow (ADR-0009) before they actually run.

## 9. Knowledge Base: Obsidian Integration

Obsidian serves as a personal, file-based knowledge base that NyxelOS accesses through the open source `obsidian-local-rest-api` plugin, which provides a local, API-key-secured REST interface as well as a built-in MCP server for the vault. This lets NyxelOS read notes, surgically patch specific sections by heading or frontmatter, and search the vault without losing control of the files â€” the vault remains a normal, local folder structure. A dedicated "docs agent" watches architecture decisions and code changes during development and automatically writes a new entry to the vault's dev-log folder for every relevant change, plus a new or updated architecture decision record (ADR) as needed. This way the knowledge base grows on its own alongside the project instead of documentation needing to be maintained after the fact. The concrete vault structure for this project already lives under `knowledge-base/` in the repository and will be kept up to date with every significant change going forward.

Beyond browsing, the knowledge base is automatically part of every model call in a workspace (ADR-0013): a compact context block â€” a full note index plus the text of always-relevant notes (`00-Meta/`) and the most recently modified notes â€” is appended to the system prompt for chats, autonomous agent runs, and scheduled automations alike, gated by a per-workspace toggle (`knowledge_base_config.inject_into_prompts`, default on). The intent is that "living project memory" is something the model always has, not something a user has to ask it to go read.

## 10. Local Data Integration: Apple and Cross-Platform

The key architectural trick here: the native companion helper is itself built as an MCP server instead of inventing a custom protocol. On the Mac, this is a small Swift menu-bar app that uses the EventKit, Contacts, and PhotoKit frameworks, triggers the normal macOS permission dialogs (TCC), and then offers tools like `calendar.list_events`, `contacts.search`, or `photos.search` as MCP tools locally â€” only on `localhost`, with its own token. NyxelOS connects to it exactly like any other MCP server, so no custom interface for "my Mac" is needed. This is implementation focus 1. As a fallback, or for a faster first pass, a CLI/AppleScript bridge (`osascript`, `icalBuddy`) provides the same MCP tools, just at higher per-call cost and without native speed. Because both variants sit behind the same MCP tool interface, a future Windows companion (via the Windows calendar/contacts APIs or Outlook interop) can later be added following the same pattern, without needing any changes to the NyxelOS core or agent engine â€” for the server, it's simply another MCP server called "My Computer."

## 11. Self-Hosting: PC Mode and Server Mode

In PC mode, a single Docker Compose stack (or optionally a native binary) starts with SQLite as the database, no reverse proxy, a single local user account, and optional password protection in case the UI needs to be reachable on the home network. In server mode, the same server container additionally runs behind Caddy with an automatic TLS certificate for your own domain, PostgreSQL as the database, and full Better-Auth login (email, passkey, or OIDC), including multi-user management and roles. A web-based setup wizard walks through choosing between the two modes on very first launch, asks for database credentials or a SQLite path, and creates the first admin account â€” exactly the previously requested point that the database choice can be made during the installation process itself, with PostgreSQL in a Docker container as the preset recommendation.

## 12. Security Architecture

Security is not an afterthought layered on top; it is built into every decision above, but is summarized here in one place â€” this section states what is actually enforced today, distinct from what's planned. Skills and plugins run **in-process**, not in isolated worker processes: `packages/skills-sdk/src/runtime.ts` enforces each skill's declared file and network access profile for any code that goes through its `ctx.*` API (`ctx.readFile`, `ctx.fetch`, etc. â€” path traversal, absolute-path escapes, and host-allowlist bypasses are all rejected, see `runtime.test.ts`), but this is **not a sandbox against a deliberately malicious skill or plugin**: code that bypasses `ctx.*` and calls Node/Bun APIs directly (`require("node:fs")`, `child_process`, raw `fetch`) has the same access as the server process itself. One exception: the `custom_code` tool kind â€” the one place a user-supplied JavaScript string is actually executed (as opposed to a skill's own hand-written/markdown-defined `run()`) â€” runs in a dedicated, short-lived subprocess with a caller-controlled (never inherited) `env` and a `node:vm`-sandboxed global scope exposing only `input`/`ctx` (`apps/server/src/plugin-sandbox.ts`, `plugin-sandbox.test.ts`); `require`/`process`/`Bun` are undefined inside it. This is a real, tested isolation boundary for that one execution point, not a claim that the whole skill/plugin runtime is sandboxed â€” everything else (skills, plugin-contributed `SKILL.md` bodies) still runs in-process as described above. Full process/worker isolation for the rest of the runtime is tracked as follow-up work (ADR-0007) and has not shipped. See `docs/PLUGIN_SECURITY.md` for the full threat model and staged mitigation plan â€” as an interim measure, remote plugin installation and custom-code skill execution are both disabled by default in production (`ENABLE_REMOTE_PLUGIN_INSTALL`, `ENABLE_CUSTOM_CODE_SKILLS`, see `apps/server/src/feature-flags.ts`) and must be explicitly opted into. Model provider API keys, the Obsidian REST key, and MCP OAuth state (`mcpServer.oauthState`) are encrypted at rest (`packages/db/src/crypto.ts`, AES-256-GCM, key from `NYXEL_ENCRYPTION_KEY`) and only decrypted at the point a repository read returns them to server-side code â€” the tRPC layer additionally strips every raw secret before a row ever reaches the browser (`toClientSafeInstallation`/`toClientSafeMcpServer`/`toClientSafeKnowledgeBaseConfig`), returning boolean flags (`hasApiKey`, `hasOAuthState`) instead. Every autonomous or super-agentic action with irreversible character â€” deleting a file, sending a message, triggering a purchase, moving a calendar entry â€” requires explicit confirmation before execution, depending on the configured autonomy level; the approval endpoints themselves require an authenticated session scoped to the workspace that owns the request. A complete, immutable audit log records every tool execution with timestamp, agent, model, and result, viewable from the UI. Local data such as calendar, contacts, or photos never leaves the device automatically; it is only shared with a cloud model when the user explicitly allows it for that specific request. For server mode, standard web hardening measures apply on top: an allowlist-based CORS policy that fails closed for unrecognized origins, per-route rate limiting (auth, tRPC, library, chat stream), a global request body size limit, and a production boot guard that refuses to start with a missing, known-weak, or too-short `BETTER_AUTH_SECRET`/`NYXEL_ENCRYPTION_KEY`. Signed plugin manifests, so that only vetted extensions are trusted automatically, remain **planned, not shipped** â€” see the Phased Roadmap and `docs/PLUGIN_SECURITY.md`'s staged mitigation plan (install-time trust warning, ref-pinning, static scanning, then real process isolation) for what's cheap-and-safe to add next versus what requires the plugin runtime's isolation work.

## 12a. Reliability Foundation

This section documents the reliability-hardening pass applied on top of the systems described above â€” what changed, and why, distinct from the rest of this document which describes steady-state behavior.

**Typed run outcomes, not text markers.** A durable task run used to decide whether it was paused (waiting on a user question, waiting on an approval, or blocked by an autonomy budget) by checking whether the model's own final answer text happened to contain the literal strings `"pending_question"` or `"pending_approval"` â€” fragile, since nothing guarantees a model echoes a tool result back verbatim. `packages/core-agent-engine/src/run-outcome.ts` replaces this: `ask_user_question` and every approval-deferring tool call record a structured `PauseReason` directly onto a per-run `RunOutcomeSignal` at the moment they defer, and `agent-runtime.ts`/`scheduler.ts` read that signal (and the already-structured autonomy-budget block reason) back via `resolveRunOutcome()` instead of pattern-matching text. Priority when more than one fired in the same turn â€” question, then approval, then budget â€” mirrors the pre-existing behavior exactly.

**Centralized tool permission policy.** `packages/core-agent-engine/src/permissions.ts` adds a `ToolPermissionDescriptor` (`source`, `toolKind`, `category`, `riskLevel`, `sideEffects`, `requiredScopes`, `requiresApproval`, `sandboxRequired`) resolved by `resolveToolPermission()` â€” the single place every tool source (skills, workspace tools, MCP tools, workflow tools, delegation) can ask "what is this call and does it need approval/sandboxing," rather than each source re-deriving the answer. `ALWAYS_REQUIRES_APPROVAL_KINDS` (terminal execution, custom code, file deletion, Playwright code â€” see section 12) now lives here too, so it can't drift between call sites. The `run_workflow` and `delegate_to_agent` tools, which previously wrote audit log entries with no permission snapshot at all, now attach one through this same descriptor.

**Super-agent delegation depth and cycle protection.** A super-agent could previously delegate to another super-agent that delegates back to it â€” `ctx.allowDelegation` only gated the model-driven `delegate_to_agent` tool, and the planner-driven auto-delegation branch (`agent-runtime.ts`) had no depth or cycle check of its own at all, so a misconfigured delegate whitelist could recurse without bound. `packages/core-agent-engine/src/delegation-policy.ts` adds `MAX_DELEGATION_DEPTH` (3) and `filterCyclicCandidates()`; both delegation paths now thread a depth counter and an ancestor-agent-id chain through every recursive `executeManagedTask` call, refuse to expose/enter delegation past the max depth, and drop any candidate already in the chain even if it's still on the whitelist. The planner's execution plan also now asks for a `delegationTasks: {agentId, instruction}[]` â€” a distinct sub-instruction per delegate â€” instead of handing every delegated child the parent task's instruction verbatim; a best-effort, non-blocking verifier call checks the synthesized final answer against the plan's own completion check and records the result on the task timeline. Each child's outcome (success/failure, task id, result summary) is persisted onto the parent task's `plan` field as `delegationResults`.

**Durable execution and retries.** `agentRun` gained `retryCount`, `maxRetries` (default 3), and `nextRetryAt` columns, plus a `dead_letter` terminal status. A thrown error from a run's own execution (not a budget pause, not a tool error the model already saw) is classified as transient (network/timeout/rate-limit/5xx-shaped) or not; a transient failure retries in place â€” same task/run rows, same lease â€” with capped exponential backoff up to `maxRetries`, landing in `dead_letter` once exhausted, or `failed` immediately for a non-transient error. Previously, a thrown execution error was only ever recorded as a task/run failure by whichever caller happened to catch it (`scheduler.ts`'s automation path did; a chat- or delegate-triggered failure did not, leaving the run stuck at `running` until the ~90s stale-run sweep noticed the expired lease) â€” `executeManagedTask` now records the terminal state itself, for every trigger, before rethrowing.

**Secret redaction extended to the task timeline.** The audit log has always redacted secret-shaped keys and capped serialized size (`apps/server/src/audit.ts`); task-event payloads for tool calls/errors (visible in the Tasks UI, not just the audit log) previously only truncated, not redacted. They now go through the same `sanitizeForAudit()` before truncation.

**Sandboxed custom-code output cap.** `apps/server/src/plugin-sandbox.ts`'s isolated `custom_code` execution boundary (section 12) now caps both the returned value and any thrown error message to 200KB serialized by default, so a runaway script can't balloon the caller's memory/DB row/model context just because the process/vm isolation itself succeeded.

## 13. UI/UX and Design System

The UI adheres without exception to shadcn/ui in its default theme with the default color variables from globals.css â€” no custom color palette, no individual branding color scheme, to preserve consistency across every area. The key areas are the chat area with streaming responses and visible tool calls, an agent dashboard for managing all agents and their autonomy levels, a marketplace for skills and plugins, an overview of connected MCP servers, model provider management, an embedded view of the knowledge base including an Obsidian graph view, an automation monitor for running and scheduled autonomous agents, and the audit log. shadcn/ui released its own chat-interface components in June 2026, which serve as the starting point for the streaming chat area instead of rebuilding chat bubbles from scratch.

## 14. Streaming Architecture

Responses are streamed token by token from the Bun/Hono server to the frontend via server-sent events. TanStack Query itself does not manage the streaming connection; instead, a dedicated hook updates the query cache on every received chunk event, so the rest of the screen (sidebar, agent list) stays in sync normally through TanStack Query. Tool calls that happen mid-stream are rendered inline in the chat history as their own immediately visible cards, before the model continues with the actual text response.

## 17. Open Questions and Next Steps

Before a public launch, the name "NyxelOS" should be checked once more against domain availability, npm package name, and existing trademarks, even though the research showed no collision. Technically open questions also remain: the exact sandbox technology for skills (e.g. Bun's own workers vs. lightweight containers), whether a single Docker Compose stack is enough for server mode or whether a Kubernetes-capable variant should be added, and the priority between Phase 4 (Apple companion) and Phase 3 (Obsidian) if both at once would be too much. The next sensible step would be to start Phase 0 as a standalone work package: set up the monorepo, get the database layer working with both dialects, and show a first streaming chat against a local Ollama model.
