# NyxelOS — Architecture Plan for a Self-Hosted, Open Source agentic OS

Date: July 1, 2026 · Status: Planning phase, no code written yet

## 1. Vision and Guiding Principles

NyxelOS is a fully open source operating system for AI agents that runs entirely on your own computer or your own server and never sends data to third parties unless the user deliberately connects an external model provider. The name is a short, invented, infrastructure-sounding word with no literal meaning attached — chosen deliberately as a clean, brandable product name rather than a descriptive compound. A collision check found no existing open source projects, products, or notable accounts under this name, while more literal options like "AgentOS" are already taken by several projects (e.g. agentos.org, The-Swarm-Corporation/AgentOS, rivet-dev/agentos), and other invented candidates considered along the way (Hearthloom, Vantrix, Halvex, Draveon) were either dropped in favor of a cleaner sound or turned out to collide with existing small projects. If "NyxelOS" turns out to collide during a final domain/trademark check, "Quorin" or "Ilvara" are backup names — both also came up empty in research.

Four principles run through every decision in this plan. First, local-first: the core functionality must work without an internet connection and without external accounts once a local model is installed. Second, modularity: models, skills, MCP servers, plugins, and even the database are swappable building blocks behind clear interfaces, never hard-wired. Third, security by default: every autonomy level an agent has must be explicitly granted; nothing should silently gain more permissions than the user allowed. Fourth, one consistently applied design: shadcn/ui in its default theme, exclusively the default color tokens from globals.css, no custom colors, no deviation from the component library.

## 2. Architecture Overview

NyxelOS consists of five major layers. The web UI (React, TanStack Router, TanStack Query, shadcn/ui) talks exclusively to the backend server, never directly to models or external services. The backend server (Bun runtime, Hono as the HTTP layer, tRPC for typed requests, dedicated SSE routes for streaming) handles authentication, data storage, and the actual agent engine. The agent engine orchestrates chats, autonomous agents, and super-agents, picks models from the model provider layer, and calls tools through the MCP client. The MCP layer is bidirectional: NyxelOS connects as a client to any MCP server (Obsidian, filesystem, browser, your own Mac) while also exposing its own MCP server, through which skills can be offered outward (e.g. to Claude Desktop or Claude Code). The fifth layer is the local companion helpers — small native processes on the respective operating system that expose calendar, contacts, and photos without the NyxelOS server itself needing OS-native permissions.

This separation enables exactly the two required deployment modes: on your own PC, everything including the database runs in a single process or Docker Compose stack without internet access; on your own server, the same server container runs behind a reverse proxy with a real domain and login, without any code changes.

## 5. Domain Model

At the center is the workspace, a category into which users sort their chats, agents, and automations (e.g. "Work", "Personal", "Coding", "Research"). Each workspace has its own custom instructions, which are prepended as a system-prompt block before every conversation, as well as its own selection of enabled skills and MCP servers. A chat is a single conversation with a chosen model; an agent is a saved configuration of system prompt, model, assigned skills/MCP tools, and an autonomy level. Autonomy levels are tiered: "Chat" only responds on request, "Assisted" may call tools but must have the user confirm any security-relevant action (deleting a file, sending a message, triggering a purchase), "Autonomous" runs on a schedule or in response to events in the background within previously granted limits, and "Super-agent" can additionally delegate to other agents as subtasks and merge their results. Every action by every agent is logged in the audit log, including the model used, the tool called, and the result.

## 6. The Agent System in Detail

Normal chats are the simplest case: a user types, the chosen model responds with streaming, optionally using individual skills or MCP tools the user has enabled for that chat. Autonomous agents run without active input, triggered either by a schedule (e.g. "summarize today's calendar every morning at 7am") or by an event (e.g. "when a new file lands in the Downloads folder"). They have a tightly scoped tool and network budget defined at creation time, and can request approval before performing an irreversible action. Super-agents are orchestrators: they break a complex request into subtasks, assign each subtask to a specialized sub-agent or skill, collect the intermediate results, and merge them — conceptually similar to a project lead coordinating specialists, except that each "specialist" here can be its own configured agent with its own model and tools.

## 7. Model Provider Layer

NyxelOS distinguishes three provider categories, all addressed through the same Vercel AI SDK abstraction. Local models are auto-detected on startup by having the server probe common local ports: Ollama at `localhost:11434/api/tags`, LM Studio and other OpenAI-compatible servers at `/v1/models`. Detected models appear automatically in the model picker with no manual configuration. Cloud API models (Anthropic, OpenAI, Google, others) are connected via stored API keys, which are held encrypted in the database. As a third category, any OpenAI-compatible endpoint can be added manually, which also covers self-hosted third-party models or data-center clusters. For every model, the UI shows context window size, whether it runs locally or externally, and for API models, a rough cost estimate per request.

## 8. Skills, Plugins, and MCP Servers

Skills are small, self-contained capabilities (e.g. "summarize a PDF," "create an Obsidian note") that run in a sandboxed runtime and carry a clearly declared permission profile (filesystem access, network access, which directories). Plugins are larger extensions that can bundle multiple skills, their own UI panels, and their own background processes — comparable to an app-store listing. MCP servers are the third, complementary building block: instead of building every integration from scratch, NyxelOS connects to the growing MCP ecosystem (per the official MCP registry and community directories, now tens of thousands of available servers), for example for GitHub, filesystem, browser automation, or Obsidian itself. The MCP specification finalizing in July 2026 introduces a stateless protocol core, caching metadata for tool listings, and stricter authorization, all of which the MCP client layer is designed around from the start. Every skill, plugin, and MCP server goes through the same permission dialog before activation, which the user must explicitly confirm.

## 9. Knowledge Base: Obsidian Integration

Obsidian serves as a personal, file-based knowledge base that NyxelOS accesses through the open source `obsidian-local-rest-api` plugin, which provides a local, API-key-secured REST interface as well as a built-in MCP server for the vault. This lets NyxelOS read notes, surgically patch specific sections by heading or frontmatter, and search the vault without losing control of the files — the vault remains a normal, local folder structure. A dedicated "docs agent" watches architecture decisions and code changes during development and automatically writes a new entry to the vault's dev-log folder for every relevant change, plus a new or updated architecture decision record (ADR) as needed. This way the knowledge base grows on its own alongside the project instead of documentation needing to be maintained after the fact. The concrete vault structure for this project already lives under `knowledge-base/` in the repository and will be kept up to date with every significant change going forward.

## 10. Local Data Integration: Apple and Cross-Platform

The key architectural trick here: the native companion helper is itself built as an MCP server instead of inventing a custom protocol. On the Mac, this is a small Swift menu-bar app that uses the EventKit, Contacts, and PhotoKit frameworks, triggers the normal macOS permission dialogs (TCC), and then offers tools like `calendar.list_events`, `contacts.search`, or `photos.search` as MCP tools locally — only on `localhost`, with its own token. NyxelOS connects to it exactly like any other MCP server, so no custom interface for "my Mac" is needed. This is implementation focus 1. As a fallback, or for a faster first pass, a CLI/AppleScript bridge (`osascript`, `icalBuddy`) provides the same MCP tools, just at higher per-call cost and without native speed. Because both variants sit behind the same MCP tool interface, a future Windows companion (via the Windows calendar/contacts APIs or Outlook interop) can later be added following the same pattern, without needing any changes to the NyxelOS core or agent engine — for the server, it's simply another MCP server called "My Computer."

## 11. Self-Hosting: PC Mode and Server Mode

In PC mode, a single Docker Compose stack (or optionally a native binary) starts with SQLite as the database, no reverse proxy, a single local user account, and optional password protection in case the UI needs to be reachable on the home network. In server mode, the same server container additionally runs behind Caddy with an automatic TLS certificate for your own domain, PostgreSQL as the database, and full Better-Auth login (email, passkey, or OIDC), including multi-user management and roles. A web-based setup wizard walks through choosing between the two modes on very first launch, asks for database credentials or a SQLite path, and creates the first admin account — exactly the previously requested point that the database choice can be made during the installation process itself, with PostgreSQL in a Docker container as the preset recommendation.

## 12. Security Architecture

Security is not an afterthought layered on top; it is built into every decision above, but is summarized here in one place. Skills and plugins run in isolated worker processes with their own declared file and network access profile, so a faulty or malicious skill cannot reach the rest of the system. API keys and other secrets sit encrypted in the database and are only decrypted in memory at runtime. Every autonomous or super-agentic action with irreversible character — deleting a file, sending a message, triggering a purchase, moving a calendar entry — requires explicit confirmation before execution, depending on the configured autonomy level. A complete, immutable audit log records every tool execution with timestamp, agent, model, and result, viewable from the UI. Local data such as calendar, contacts, or photos never leaves the device automatically; it is only shared with a cloud model when the user explicitly allows it for that specific request. For server mode, standard web hardening measures apply on top: CSRF and CORS protection, per-user rate limiting, minimal necessary OAuth scopes for every third-party connection, and signed manifests for plugins so that only vetted extensions are trusted automatically.

## 13. UI/UX and Design System

The UI adheres without exception to shadcn/ui in its default theme with the default color variables from globals.css — no custom color palette, no individual branding color scheme, to preserve consistency across every area. The key areas are the chat area with streaming responses and visible tool calls, an agent dashboard for managing all agents and their autonomy levels, a marketplace for skills and plugins, an overview of connected MCP servers, model provider management, an embedded view of the knowledge base including an Obsidian graph view, an automation monitor for running and scheduled autonomous agents, and the audit log. shadcn/ui released its own chat-interface components in June 2026, which serve as the starting point for the streaming chat area instead of rebuilding chat bubbles from scratch.

## 14. Streaming Architecture

Responses are streamed token by token from the Bun/Hono server to the frontend via server-sent events. TanStack Query itself does not manage the streaming connection; instead, a dedicated hook updates the query cache on every received chunk event, so the rest of the screen (sidebar, agent list) stays in sync normally through TanStack Query. Tool calls that happen mid-stream are rendered inline in the chat history as their own immediately visible cards, before the model continues with the actual text response.

## 16. Recommended Custom Skills for Development

For the actual implementation, it's worth setting up a few project-specific Claude skills instead of re-explaining how this project works every time. Useful candidates would be a skill that always generates new shadcn components in the specified default theme and automatically checks them against globals.css, a skill for scaffolding new MCP servers following the pattern established here (tool definition, permission profile, registration), a skill that generates and tests Drizzle migrations for Postgres and SQLite simultaneously, and the previously mentioned docs-agent skill that automatically logs code changes into `knowledge-base/`. These can be set up with the skill-creator tool once the first code foundations are in place.

## 17. Open Questions and Next Steps

Before a public launch, the name "NyxelOS" should be checked once more against domain availability, npm package name, and existing trademarks, even though the research showed no collision. Technically open questions also remain: the exact sandbox technology for skills (e.g. Bun's own workers vs. lightweight containers), whether a single Docker Compose stack is enough for server mode or whether a Kubernetes-capable variant should be added, and the priority between Phase 4 (Apple companion) and Phase 3 (Obsidian) if both at once would be too much. The next sensible step would be to start Phase 0 as a standalone work package: set up the monorepo, get the database layer working with both dialects, and show a first streaming chat against a local Ollama model.
